**Required nodes (use these exact IDs / names so Plan 03 verification can grep them):**

1. **Webhook node** — `name: "Webhook social-publish"`, `path: "social-publish"`, `httpMethod: POST`, `responseMode: onReceived`, `responseCode: 200`. Outputs `{post_id}` from `body.post_id`.

3. **Postgres node "Claim Pending Rows"** — connected from Schedule Trigger ONLY. Use credential id matching the existing campaign-blast Postgres credential. Operation: Execute Query. Query EXACTLY:
   ```sql
   UPDATE social_posts
   SET status = 'publishing', updated_at = now()
   WHERE id IN (
       SELECT id FROM social_posts
       WHERE status = 'scheduled' AND scheduled_at <= now()
       ORDER BY scheduled_at ASC
       LIMIT 20
       FOR UPDATE SKIP LOCKED
   )
   RETURNING id, caption, image_url, platforms, campaign_id;
   ```
4. **Postgres node "Load Single Post"** — connected from Webhook node ONLY. Operation: Execute Query. Query EXACTLY:

   UPDATE social_posts
   SET status = 'publishing', updated_at = now()

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Internet → Caddy `/uploads/*` | Public HTTPS fetch of marketing images by Meta servers (and anyone who knows a UUID) |
| n8n → Meta Graph API | Outbound calls carry the Page Access Token |
| Streamlit → n8n webhook | Internal Docker network only — never exposed by Caddy |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-07 | Tampering / XSS | Caddy `/uploads/*` static serving | mitigate | `X-Content-Type-Options: nosniff` header forced; `Content-Disposition: inline` only for image MIME (extension allowlist enforced upstream by Plan 01); files cannot be HTML/JS by extension policy |
| T-06-08 | Information Disclosure | UUID image URLs guessable | accept | UUIDv4 has ~122 bits entropy — practically unguessable. Marketing content (not PHI) per CLAUDE.md. Acceptable. |
| T-06-09 | Information Disclosure | Meta Page Access Token leakage in n8n logs | mitigate | Token stored as n8n credential (NOT in workflow JSON); referenced via `$env.META_PAGE_ACCESS_TOKEN` in form bodies; n8n masks credentials in execution logs by default. Workflow JSON committed to git contains no token literals. |
| T-06-10 | DoS | Dispatcher mass-posting after compromise | mitigate | `LIMIT 20` per dispatcher run × 1 minute = max 1200/hr theoretical; real volume <10/day. Hard cap is enforced in SQL. |
| T-06-11 | Repudiation | Webhook replay from LAN | mitigate | n8n `/webhook/*` only listens on internal Docker network (`http://n8n:5678`); Caddy does NOT proxy `/webhook/*` to the public internet. Verified by absence of webhook route in Caddyfile. |
| T-06-12 | Tampering | Race condition: dispatcher fires same row twice | mitigate | `FOR UPDATE SKIP LOCKED` + status transition to `publishing` inside the SELECT-UPDATE atomic statement — second run finds zero rows. (RESEARCH Pitfall 5) |
| T-06-13 | Tampering | SQL injection via post_id from webhook | mitigate | post_id cast to `::uuid` — Postgres rejects malformed UUIDs before query execution. |
| T-06-14 | Spoofing | Forged webhook from inside docker network | accept | All containers on the `clinic-crm` docker network are trusted (PROJECT.md single-tenant constraint); webhook auth deferred to v2 multi-tenant. |
</threat_model>
