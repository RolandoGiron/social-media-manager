    **Create `n8n/workflows/sub-booking-flow.json`:**

    This sub-workflow handles the multi-turn booking conversation. It receives the current conversation context and message, processes the current booking step, and returns updated context + response text.

    **Inputs** (from Execute Workflow Trigger node):
    - `message_text`: patient's current message
    - `booking_context`: the context JSONB from the conversation (contains booking_step, service_type, etc.)
    - `patient_id`: UUID or null
    - `patient_name`: from patients table or from pushName
    - `wa_contact_id`: for linking to patient
    - `remote_jid`: WhatsApp JID for calendar event description
    - `instance_name`: Evolution API instance
    - `api_key`: Evolution API key

    **Flow by booking_step:**

    **Step: awaiting_service_type** (first turn after intent=BOOKING)
    - Node: OpenAI Chat node to extract service type from free text (per D-06, natural conversation)
      - System prompt: "Extract the type of dermatology service the patient wants. Common types: Consulta general, Limpieza facial, Tratamiento de acne, Revision de lunares, Botox, Peeling quimico. If the patient hasn't specified a service, respond with 'NEEDS_INPUT'. If they specified a service, respond with ONLY the service name."
      - User message: `{{ $json.message_text }}`
    - IF node: result == 'NEEDS_INPUT'?
      - TRUE: Return response "Que tipo de servicio te gustaria? Ofrecemos consulta general, limpieza facial, tratamiento de acne, revision de lunares, entre otros." Keep booking_step as awaiting_service_type.
      - FALSE: Save service_type in context, advance booking_step to `awaiting_datetime`. Return response "Perfecto, {service_type}. Que fecha y hora te gustaria? Por ejemplo: 'manana a las 10am' o 'el jueves 17 de abril a las 3pm'."

    **Step: awaiting_datetime**
    - Node: OpenAI Chat node to extract date/time from free text
      - System prompt: "Extract the preferred appointment date and time from the patient's message. Today is {{ $now.format('yyyy-MM-dd') }}. Return a JSON object: {\"date\": \"YYYY-MM-DD\", \"time\": \"HH:mm\"}. If the patient hasn't given enough information, return {\"error\": \"NEEDS_INPUT\"}."
      - Temperature: 0.1
    - IF node: result contains error?
      - TRUE: Return response "No entendi la fecha. Puedes darme un dia y hora? Por ejemplo: 'el lunes a las 10am'."
      - FALSE: Parse date/time from LLM response
    - **Google Calendar node: Get Events** (check availability):
      - Calendar ID: from environment variable or n8n expression (CLINIC_CALENDAR_ID)
      - Time Min: parsed date at start of business hours (9:00)
      - Time Max: parsed date at end of business hours (18:00)
      - Operation: Get Many events
    - **Code/Function node: Find available slots**:
      - Take the existing events and find 2-3 open 30-minute slots near the patient's preferred time
      - If no slots on that day: find slots on the next business day
      - Output: array of available slot objects `[{"start": "2026-04-15T10:00:00", "end": "2026-04-15T10:30:00"}, ...]`
    - IF node: slots found?
      - TRUE: Save available_slots in context, advance to `confirming_slot`. Return response listing the options:
        "Tengo estos horarios disponibles:\n1. {slot1_formatted}\n2. {slot2_formatted}\n3. {slot3_formatted}\nCual prefieres? (responde con el numero)"
      - FALSE: Return "No hay horarios disponibles para esa fecha. Quieres intentar otro dia?" Keep step as awaiting_datetime.

    **Step: confirming_slot**
    - Parse patient's selection (number 1, 2, or 3 from message)
    - IF node: valid selection?
      - TRUE:
        1. **Google Calendar node: Create Event**
           - Calendar ID: CLINIC_CALENDAR_ID
           - Summary: "{service_type} - {patient_name}"
           - Start: selected slot start time
           - End: selected slot end time
           - Description: "Paciente: {patient_name}, WhatsApp: {wa_contact_id}"
        2. **Postgres node: Insert appointment**
           ```sql
           INSERT INTO appointments (patient_id, appointment_type, scheduled_at, google_event_id, duration_minutes, notes)
           VALUES ('{{ $json.patient_id }}', '{{ $json.booking_context.service_type }}', '{{ selected_slot.start }}', '{{ $json.google_event_id }}', 30, 'Agendada via chatbot')
           RETURNING *
           ```
           NOTE: If patient_id is null (unknown patient), the workflow should attempt to find or create the patient first. For v1, if patient is unknown, set booking_step back and ask for name, then create patient in patients table before booking.
        3. Set booking_step to `complete`, state back to `awaiting_intent`
        4. Return confirmation message (per CONTEXT.md specifics, CAL-02):
           `"Cita confirmada para el {fecha_formatted} a las {hora_formatted}. Te esperamos en la clinica. Hasta pronto!"`
      - FALSE: Return "No entendi tu seleccion. Por favor responde con 1, 2 o 3."

    **Outputs** (returned to main workflow):
    - `response_text`: the bot's reply to send
    - `new_state`: updated conversation state (booking_flow or awaiting_intent if complete)
    - `new_context`: updated context JSONB with booking progress

    **Update `n8n/workflows/whatsapp-chatbot.json`:**
    - In the Switch node's BOOKING branch: replace the placeholder Set node with an Execute Workflow node calling sub-booking-flow
    - When conversation state is already `booking_flow`: route directly to Execute Workflow node for sub-booking-flow (bypass intent classification since we know the patient is mid-booking)
    - Pass inputs: message_text, booking_context (from conversation.context), patient_id, patient_name, wa_contact_id, remote_jid, instance_name, api_key
    - After sub-booking-flow returns: use its outputs for state update and message send
