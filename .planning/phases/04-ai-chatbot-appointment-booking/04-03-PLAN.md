    **Workflow flow (per RESEARCH.md architecture and D-10/D-12):**

    1. **Webhook Trigger node** — path: `whatsapp-inbound/messages-upsert` (same path as the stub it replaces)
       - Method: POST
       - Receives Evolution API MESSAGES_UPSERT payload

    2. **IF node: Filter fromMe** — Check `{{ $json.data.key.fromMe }}` equals false
       - TRUE branch continues processing
       - FALSE branch stops (prevents infinite bot loop — Pitfall 3 from RESEARCH.md)

    3. **Set node: Extract fields** — Extract useful fields into clean variables:
       - `remote_jid`: `{{ $json.data.key.remoteJid }}`
       - `message_text`: `{{ $json.data.message.conversation }}` (fallback to `{{ $json.data.message.extendedTextMessage.text }}` for quoted replies)
       - `push_name`: `{{ $json.data.pushName }}`
       - `wa_message_id`: `{{ $json.data.key.id }}`
       - `instance_name`: `{{ $json.instance }}`

    4. **Postgres node: Read conversation state** — Query:
       ```sql
       SELECT c.id, c.state, c.context, c.patient_id, c.wa_contact_id,
              p.first_name, p.last_name, p.phone_normalized
       FROM conversations c
       LEFT JOIN patients p ON c.patient_id = p.id
       WHERE c.wa_contact_id = '{{ $json.remote_jid }}' AND c.state != 'closed'
       ORDER BY c.last_message_at DESC
       LIMIT 1
       ```
       Operation: Execute Query (so FOR UPDATE can be used if supported; otherwise rely on optimistic approach)

    5. **IF node: Conversation exists?** — Check if Postgres returned rows
       - FALSE branch: **Postgres node: Create new conversation** — INSERT into conversations (wa_contact_id, state='new'). Also try to link patient_id by matching `remote_jid` phone part against patients.phone_normalized.
       - TRUE branch: continues with existing conversation

    6. **Merge node** — Merge new/existing conversation data into single flow

    7. **Postgres node: Insert inbound message** — INSERT into messages (conversation_id, direction='inbound', sender='patient', content=message_text, wa_message_id)

    8. **IF node: State is human_handoff?** — If conversation state == 'human_handoff':
       - TRUE: Skip bot processing entirely (message was stored in step 7, admin will see it in inbox). End workflow.
       - FALSE: Continue to bot processing.

    9. **Postgres node: Load knowledge base** — `SELECT pregunta, respuesta, categoria FROM knowledge_base WHERE is_active = true ORDER BY categoria`
       - Format results into a string for LLM context: `Q: {pregunta}\nA: {respuesta}\n---\n` for each row.

    10. **Code/Function node: Format FAQs** — Transform KB rows into formatted string for LLM prompt injection.

    11. **Execute Workflow node: Classify intent** — Call sub-classify-intent with inputs: `message_text`, `formatted_faqs`

    12. **Switch node: Route by intent** — Based on classify-intent output:
        - `FAQ` → Execute Workflow node: Call sub-faq-answer with `message_text`, `formatted_faqs`
        - `BOOKING` → Set node: Set response to "Claro, con gusto te ayudo a agendar una cita. Que tipo de servicio necesitas?" and update state to `booking_flow` (booking sub-workflow built in Plan 04)
        - `HANDOFF` → Set node: Set response to "Tu consulta fue recibida. Un miembro de nuestro equipo te atendera en breve." (per CONTEXT.md specifics) and update state to `human_handoff`
        - `GREETING` → Set node: Set response to "Hola! Soy la asistente virtual de la clinica. En que puedo ayudarte? Puedo responder preguntas sobre nuestros servicios, horarios, ubicacion, o ayudarte a agendar una cita."
        - Default → Same as HANDOFF

    13. **Merge node** — Merge all intent branches back together

    14. **IF node: Check for handoff trigger in FAQ response** — If sub-faq-answer response contains "No tengo esa informacion. Te comunico con un miembro de nuestro equipo", set state to `human_handoff` (per BOT-02)

    15. **Postgres node: Update conversation state** — UPDATE conversations SET state=new_state, context=updated_context, last_message_at=now() WHERE id=conversation_id

    16. **Postgres node: Insert bot response message** — INSERT into messages (conversation_id, direction='outbound', sender='bot', content=response_text)

    17. **Execute Workflow node: Send WA message** — Call sub-send-wa-message with `remote_jid`, `response_text`, `instance_name`, `api_key` (api_key from env/credential)

    **State transitions implemented:**
    - new/awaiting_intent + FAQ intent → faq_flow → (after answer) → awaiting_intent
    - new/awaiting_intent + BOOKING intent → booking_flow (handled in Plan 04)
    - new/awaiting_intent + HANDOFF intent → human_handoff
    - new/awaiting_intent + GREETING → awaiting_intent
    - faq_flow + response contains handoff phrase → human_handoff
    - human_handoff → messages stored, no bot response
    - closed + new message → new (reopen)

    **Environment variables used in workflow:**
    - Evolution API URL: `http://evolution-api:8080` (Docker internal)
    - Evolution API key: from n8n credential or environment
    - OpenAI API key: from n8n credential (per D-02)

    **IMPORTANT:** The booking_flow branch in the Switch node should set a placeholder response for now. The actual multi-turn booking logic is built in Plan 04 (sub-booking-flow.json). For this plan, when intent=BOOKING and state transitions to booking_flow, send the initial greeting and set context to `{"booking_step": "awaiting_service_type"}`.
